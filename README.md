# pi-tdd

Guided TDD extension for the [Pi coding agent](https://github.com/earendil-works/pi-coding-agent). Behavioral agents (architect, red-writer, green-impl) + RED→GREEN gates + cycle-locked flow state.

**Generic & portable.** Any monorepo — Python, TypeScript, or polyglot. Not tied to any framework or project. Configure services via `.pi/tdd-services.json`.

## Philosophy

TDD needs **different mindsets**: RED (write tests that fail) and GREEN (minimum code to pass). One agent doing both over-implements. This extension splits them into **three behavioral agents**:

| Agent | Role | Never |
|---|---|---|
| `architect` | Recon + plan | Implements or writes tests |
| `red-writer` | Failing tests + stubs | Implements production code |
| `green-impl` | Minimum implementation | Touches tests |

Skills (passed at call time via `tdd-services.json`) provide **service specialization** — one agent roster serves all services in the monorepo.

## Quick start

```bash
# 1. Install
pi install pi-tdd

# 2. Scaffold
/tdd:init              # analyzes project, creates config + agents

# 3. Start a cycle
tdd_start({ service: "api", feature: "user auth" })

# 4. Follow the flow
tdd_next()             # get next step instructions
tdd_red({...})         # verify tests FAIL
tdd_green({...})       # verify tests PASS
tdd_done()             # release lock
```

## Commands & Tools

All prefixed `tdd_` for zero collision risk.

| Name | Type | What |
|---|---|---|
| `/tdd:init` | Command | Analyze project, scaffold config + agents, report gaps |
| `tdd_start` | Tool | Start a new locked TDD cycle |
| `tdd_next` | Tool | Get the next phase's subagent call |
| `tdd_red` | Tool | RED gate — run tests, verify they FAIL, write manifest |
| `tdd_green` | Tool | GREEN gate — run tests, verify they PASS (max 100 retries) |
| `tdd_status` | Tool | Show active cycle details |
| `tdd_done` | Tool | Release cycle lock, mark complete or abandoned |

## Project config: `.pi/tdd-services.json`

Generated by `/tdd:init`, edit manually for full control:

```json
{
  "api": {
    "dir": "services/api",
    "skill": "api-tech",
    "runner": "pytest",
    "cmdTemplate": "poetry run pytest {paths} -q"
  },
  "web": {
    "dir": "apps/web",
    "skill": "web-tech",
    "runner": "vitest",
    "cmdTemplate": "pnpm exec vitest run {paths} --reporter=json"
  }
}
```

- `dir` — path relative to repo root
- `skill` — skill name passed to subagents (create `.pi/skills/<skill>/SKILL.md`)
- `runner` — `pytest`, `vitest`, or `jest`
- `cmdTemplate` — `{paths}` is replaced with space-joined test file paths

## Requirements

- **Pi coding agent** with subagent support
- **Git repository** (worktree root detection)
- At least one **test framework**: pytest, vitest, or jest
- **pyproject.toml** (pytest) or **package.json** (vitest/jest) in each service dir with test dependencies installed

## Flow (6 phases)

```
tdd_start       →  📐 architect      (recon + plan)
tdd_next        →  🔴 red-writer     (failing tests + stubs)
tdd_red         →  🔴✅ red_verify   (verify tests FAIL)
                    🟢 green-impl     (minimum implementation)
tdd_green       →  🟢✅ green_verify (verify tests PASS)
                    👀 reviewer       (code review)
tdd_done        →  ✅ done           (lock released)
```

Each cycle gets a unique 6-char ID, isolated in `.tdd/<service>/<id>/`. Only one cycle active at a time.

## Widget

The extension shows a live widget below the editor during TDD:

```
🧪 TDD · api/a3f2b1 🔒
📐 architect · user auth
```

Lock indicator: 🔒 active, 🔓 completed. Refreshes immediately on state changes (`tdd_start`, `tdd_done`).
